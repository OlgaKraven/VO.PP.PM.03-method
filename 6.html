6.1. Принципы итеративной разработки (Agile / Scrum)

6.1.1. Понятие

Итеративная разработка — это процесс создания веб-приложений, при котором проект развивается поэтапно (итерациями), с регулярным получением обратной связи и корректировкой решений. Основу подхода составляют гибкие методологии — Agile и Scrum.

Цель: уменьшить риски, повысить гибкость и обеспечить постоянное улучшение продукта на основе пользовательского опыта.

6.1.2. Принципы Agile

ПринципСутьПрименение в веб-разработке
ИтеративностьРазработка ведётся короткими цикламиСоздание MVP, последующая доработка функций
Обратная связьПостоянное взаимодействие с заказчиком и пользователемТестирование прототипов, UX-аудит
ГибкостьВозможность изменить требования в процессеКорректировка интерфейса после отзывов
Командная работаКоммуникация между ролямиСовместная работа в GitHub / Figma / Miro
Ориентация на ценностьВ приоритете — полезные функцииРеализация реально используемых сценариев

Пример: вместо создания всего сайта сразу, команда выпускает первую версию (MVP) с базовым функционалом («афиша + покупка билетов»), затем итеративно добавляет фильтры, отзывы, уведомления.

6.1.3. Структура Scrum-процесса

ЭлементНазначениеПример применения
Product BacklogСписок функций и задач«Добавить поиск», «Сделать адаптивную вёрстку»
Sprint BacklogЗадачи текущего цикла5 задач на неделю
Sprint (итерация)Короткий цикл разработкиМодуль «Регистрация пользователей»
Daily ScrumКороткие встречи команды15 минут: прогресс, блокеры
ReviewДемонстрация результатаПоказ прототипа страницы
RetrospectiveАнализ и улучшенияЧто получилось, что улучшить

Результат: каждая итерация завершает мини-проект, который можно протестировать и показать пользователю.

6.2. Документирование проекта

6.2.1. Назначение

Документирование — неотъемлемая часть профессиональной разработки. Оно обеспечивает понятность, воспроизводимость и поддержку проекта.

Цель: зафиксировать архитектуру, структуру данных, интерфейс, процессы и результаты тестирования.

6.2.2. Основные виды проектных документов

Вид документаНазначениеИнструменты
Отчёт о проектеОписание цели, этапов, структуры, результатовWord / PDF / Markdown
Диаграммы архитектурыВизуализация компонентов и их связейPlantUML, Miro, draw.io
ER-диаграммы БДМодель данных и связейMySQL Workbench, dbdiagram.io
Диаграммы C4 / UMLУровни системы и взаимодействие модулейMiro, Lucidchart, PlantUML
Схемы пользовательских потоков (User Flow)Пути пользователяFigma, Miro
Отчёты Lighthouse и валидаторовАудит качества и производительностиChrome DevTools, PageSpeed Insights
Чек-листы и результаты тестовФункциональное и UX-тестированиеExcel, Notion, Google Sheets

Рекомендация: для учебных проектов достаточно одного отчёта, включающего описание, диаграммы, результаты аудита и тестов.

6.2.3. Структура итогового отчёта о проекте

  1. Титульный лист и цель проекта.
  2. Описание предметной области и сценариев.
  3. Архитектура веб-приложения (C4 / ER / UML).
  4. Структура интерфейса (Figma / HTML).
  5. Методы оптимизации (Lighthouse, Web Vitals).
  6. Результаты тестирования и UX-аудита.
  7. Выводы и предложения по развитию проекта.

Совет: для публикации и хранения материалов удобно использовать GitHub Pages.

6.3. Инструменты разработчика

Современный процесс проектирования и разработки веб-приложений строится на использовании специализированных инструментов, объединяющих код, дизайн, тестирование и документацию.

Visual Studio CodeРедактор кода с плагинамиРазработка HTML/CSS/JS, Git-интеграция
Git + GitHubКонтроль версий и хостингКомандная работа, фиксация этапов
CodePen / JSFiddleОнлайн-песочницыБыстрая демонстрация примеров

Совет: фиксируйте две версии проекта: «до оптимизации» и «после оптимизации» (по результатам Lighthouse).
